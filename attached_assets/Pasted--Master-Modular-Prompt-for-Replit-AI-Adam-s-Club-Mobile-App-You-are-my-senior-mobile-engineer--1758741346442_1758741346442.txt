 Your task is to help build a mobile-first application for Adam’s Club, using strict modular coding principles.

🏔️ Business Project Introduction
Adam’s Club is a subscription-based lifestyle and experience-sharing platform.
 It is built around the idea of access over ownership — members pay a monthly fee to enjoy premium outdoor gear (such as jet skis, mountain bikes, kayaks, camping kits) and curated experiences without needing to buy or maintain them.
Vision: To make adventurous weekends and outdoor lifestyles accessible, affordable, and hassle-free for everyone.


Value Proposition: Instead of spending thousands on equipment that requires storage, upkeep, and insurance, members simply book what they need when they need it, with insurance, deposits, and waivers managed seamlessly through the app.


Target Users:


Members (subscribers): Join, explore gear/experiences, book, check in/out, and manage billing in-app.


Staff/Admins: Manage bookings, scan QR codes, capture/refund deposits, oversee waivers, and keep track of damages or delays.


Partners (future): Experience vendors who can list their offerings directly.


The mobile app must also have growth-focused features like smart popups, lead capture, referral nudges, and email/push notifications — helping to convert free users into paid members and retaining them long term.
The guiding philosophy is simplicity, modularity, and reusability:
 Every part of the system should be a self-contained piece (auth, bookings, payments, waivers, analytics, marketing), so we can replace or improve one feature without breaking others.

🔑 Modular Coding Rules (apply everywhere)
Single Responsibility: Each module should do only one job (auth handles login, booking handles reservations, waiver handles legal compliance, etc.).


Interfaces First: Define TypeScript + Zod types and contracts before writing logic.


Separation of Concerns:


UI → /components (presentational only, no IO).


Feature Logic → /features/<name> (hooks, orchestration).


Business Logic → /services (pure functions, no React, no SDKs).


External IO → /integrations (Stripe, Supabase, Email, Push, Storage).


Shared Contracts → /schemas.


State → /stores (React Query/Zustand/MMKV).


Barrel Exports: Each module has index.ts exposing only its safe, public API.


Tests: Each module must include happy path + failure case tests (mocking IO).


Guardrails: Build in small, reviewable steps. Never mass-edit or overwrite. Always summarize what changed and how to test it.


Feature Flags: All marketing/experimental features must live behind /config/flags.ts.



📂 Folder Structure (Step 0 deliverable)
/app        # Navigation & routes only (thin glue)
/components # UI widgets: cards, forms, lists, modals, banners
/features   # Each feature = self-contained module (auth, booking, payments, waiver, marketing, etc.)
/services   # Pure business logic functions (deterministic, reusable)
/integrations # Stripe, Auth, DB, Email, Push, NFC, Storage clients
/stores     # State/query cache (React Query, Zustand, MMKV)
/schemas    # Zod/TS entity types, enums, DTOs
/lib        # Utilities (dates, money, QR, guards, ICS)
/emails     # Email templates (server-triggered)
/config     # Env parsing, flags, plans, marketing copy
/tests      # Unit & integration tests (per module)
/assets     # Icons, splash screens, fonts
